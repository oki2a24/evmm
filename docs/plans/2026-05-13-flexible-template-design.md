# 柔軟なWBSマッピングシステム 設計ドキュメント

## 1. 目的
現在のスクリプトが持つ「特定のテンプレート構造への強い依存」を解消し、プロジェクトごとのエクセル構成（カラム名の変更、ヘッダー位置の変更、フェーズ数の増減）に柔軟に対応できる仕組みを構築する。

## 2. 設計思想
- **Excel-as-Master**: 人間が操作するエクセルを唯一の真実のソースとし、プログラムがそれに適応する。
- **AI-as-Translator**: AIがエクセルの構造を推論し、ユーザーとの対話を通じて「役割マッピング（JSON）」を生成・管理する。
- **Role-Based Mapping**: スクリプトは物理的な列位置や名前ではなく、「開始予定日としての役割を持つ列」という抽象化された定義を参照して動作する。

## 3. システムアーキテクチャ

### 3.1 構成要素
1.  **`wbs_structure.json`**: プロジェクトごとのエクセル構造定義ファイル。
2.  **`scripts/wbs_config_manager.py` (新設)**: 
    - エクセルをスキャンし、構造を推論する。
    - ユーザーと対話してマッピングの正当性を確認する。
    - `wbs_structure.json` を生成・更新する。
3.  **既存スクリプトのリファクタリング**:
    - `update_wbs.py`, `analyze_evm.py`, `check_wbs_integrity.py` がマッピング定義を参照するように変更。

### 3.2 マッピング構造 (wbs_structure.json)
```json
{
  "sheet_name": "WBS_EVM",
  "header_row": 2,
  "data_start_row": 3,
  "columns": {
    "common": {
      "id": "機能ID",
      "name": "機能名称"
    },
    "phases": [
      {
        "name": "作成",
        "mapping": {
          "plan_start": "開始日予定",
          "plan_end": "終了日予定",
          "plan_effort": "工数予定",
          "actual_start": "開始日実績",
          "actual_end": "終了日実績",
          "actual_effort": "工数実績",
          "progress": "進捗率(%)",
          "pv": "PV (計画値)",
          "ev": "EV (出来高)",
          "ac": "AC (実績コスト)"
        }
      }
      // ... 他のフェーズも同様の構造で繰り返される
    ]
  }
}
```

## 4. 運用フロー

### 4.1 プロジェクト初期化・構造変更時
1. AIがエクセルの全カラム名をスキャン。
2. 各「役割（Role）」に合致するカラム名を正規表現や類似度で推論。
3. ユーザーに提案し、承認を得る。
4. `wbs_structure.json` を保存。

### 4.2 通常実行時
1. スクリプトは `wbs_structure.json` を読み込む。
2. JSONに定義されたカラム名が現存するかチェック。
3. 存在しない場合、または構造に矛盾がある場合は「再マッピング」をユーザーに促す。

## 5. 線引き（制約事項）
以下の前提条件のみを維持し、それ以外は柔軟に対応する。
1. 1行に1つのタスクがあること。
2. 「機能ID」など、行を一意に特定できる列が存在すること。
3. フェーズごとの項目が、横方向のブロック（繰り返し）として構成されていること。
