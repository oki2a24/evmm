# デザインドキュメント：README.md の刷新

## 1. 目的
`README.md` を、WBS/EVM のエキスパート（上司）に向けたプロフェッショナルなプロジェクト提案書兼技術概要へと刷新する。生成AIと自動化が、現場の具体的な痛み（工数、不備、リスク予兆）をいかに解決するかを、GitHub の技術ドキュメントにふさわしい冷静なトーンで提示する。

## 2. ターゲット
- **役割**: シニアプロジェクトマネージャー / WBS・EVM エキスパート。
- **関心事**: 効率性、データの整合性、理論的裏付け（PMBOK）、アクションに繋がる洞察。
- **課題感**: 分析用シート作成の工数、エクセル計算式の破損リスク、数値からのリスク言語化の手間。

## 3. 主要な価値提案（主要機能）
以下の 3 つの柱を優先順位順に強調する。
1.  **分析エンジンの自動化 (Automated Analytics Engine)**: Python スクリプトによる複雑な EVM 分析シートの即時生成。手作業によるセットアップ時間をゼロへ。
2.  **AI インサイト生成 (AI Insight Generation)**: PMBOK 理論に基づく AI メンタリング。CPI/SPI の推移を解釈し、プロジェクトのリスクを具体的な言葉でレポート。
3.  **整合性ガード (Automated Integrity Guard)**: 「絶対的な整合性」の追求。エクセルの計算式や論理的矛盾を自動検証し、「壊れたシート」での運用を防止。

## 4. 設計思想
- **Script-as-Master**: エクセルの構造やロジックにおいて、Python スクリプトを「唯一の真実のソース (SSOT)」として扱う。
- **プロフェッショナルかつ冷静なトーン**: 過度な煽りを排除し、技術的な解決策を淡々と提示する。
- **PMBOK 準拠**: すべての AI 分析とアドバイスを、確立されたプロジェクトマネジメント理論に基づいて行う。

## 5. 構成案
1.  **タイトル & 技術サマリー**: AI 駆動型フレームワークの全体像。
2.  **課題定義 (Problem Statements)**: このエコシステムが解決する具体的な課題。
3.  **主要機能 (Key Features)**: 3 つのコアソリューションの詳細。
4.  **コア思想 (Core Philosophy)**: 「Script-as-Master」と「理論第一主義」の解説。
5.  **ディレクトリ構造**: 疎結合な管理構造（`/scripts`, `/templates`, `/projects`）の紹介。

## 6. 実装戦略
- GitHub flavored Markdown のベストプラクティスを遵守。
- 専門用語（EVM, CPI/SPI, SSOT 等）を正確に使用。
- 既存の `GEMINI.md` やプロジェクトの憲法との整合性を維持。
