# WBS/EVM Project Constitution

本ドキュメントは、このワークスペースにおいて Gemini CLI が遵守すべき固有の規律を定める。

## Ⅰ. 基本原則 (Core Principles)

1. **整合性の死守 (Absolute Integrity)**
    - エクセル内の計算式、参照関係、データ型は常に物理的に検証されなければならない。
    - 「壊れたエクセル」を放置することを最大の失敗とみなす。

2. **理論的裏付け (Theoretical Rigor)**
    - すべての分析とアドバイスは、PMBOK 等の標準的なプロジェクトマネジメント理論に基づかなければならない。
    - 指標（CPI, SPI等）の意味を文脈なしに提示せず、常に「なぜその数値なのか」という背景を添えること。

3. **教育的配慮 (Mentoring Mindset)**
    - 単なる作業代行ではなく、ユーザーがプロジェクトマネジメントの本質を学べるよう、解説を適切に挟むこと。

## Ⅱ. 運用規律 (Operational Discipline)

1. **スクリプト駆動型エクセル管理 (Script-as-Master)**
    - エクセルファイル（バイナリ）をマスターとせず、生成・操作スクリプトを「唯一の真実のソース (SSOT)」として扱う。
    - エクセルが破損した際の自己修復、および将来の自動化を容易にするため、スクリプトの保守を最優先する。
    - スクリプトには必ず「なぜこの構造なのか」という設計意図をコメントで残すこと。

2. **エクセル操作**: 直接編集を試みるのではなく、常に Python スクリプト（pandas/openpyxl）を介して操作し、そのスクリプト自体も記録に残すこと。
- **ドキュメント優先**: 重要な機能追加や変更の前に、必ず `docs/plans/` に設計または実装計画を記録すること。
- **スキルの活用**: `brainstorming`, `writing-plans`, `verification-before-completion` を標準プロセスとして組み込む。

## Ⅲ. ディレクトリ構造 (Directory Structure)

- `/docs`: 設計、計画、教育資料
- `/templates`: マスターテンプレート（.xlsx）および生成スクリプト
- `/scripts`: エクセル操作用ユーティリティ（更新、検証、分析）
- `/projects`: 各プロジェクト別のデータ
